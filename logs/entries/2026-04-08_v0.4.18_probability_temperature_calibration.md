## Decisiones tecnicas

- Se añade una capa global de `probability temperature` para suavizar o endurecer la distribucion del forecast antes de convertirla en probabilidades operables.
- La calibracion usa un `alpha`:
  - `alpha < 1.0` aplana la distribucion
  - `alpha > 1.0` la hace mas agresiva
- Se crea `run_probability_temperature_calibration.py` para aprender ese `alpha` desde auditorias resueltas:
  - lee la auditoria
  - reconstruye eventos desde snapshots live fuente
  - busca el `alpha` que minimiza `log loss`
  - desempata por `Brier` y cercania a `1.0`
- Se aplica el `alpha` aprendido a:
  - validacion live
  - backtest historico
  - backtest reciente por horizonte
- `write_forecast_policy()` ahora preserva campos no relacionados con el cutoff, para no perder:
  - `station_temperature_bias_c`
  - `probability_temperature_alpha`
  - metadatos futuros similares
- `run_station_bias_diagnostics.py` ya soporta tambien snapshots de auditoria resuelta, no solo backtests.

## Resultado de calibracion

- Comando ejecutado:
  - `venv/bin/python scripts/run_probability_temperature_calibration.py --audit-snapshot-path logs/snapshots/2026-04-08_blind_snapshot_resolution_audit.json --apply`
- Snapshot generado:
  - `logs/snapshots/2026-04-08_probability_temperature_calibration.json`
- Muestra usada:
  - `40` eventos
- Resultado seleccionado:
  - `alpha = 0.55`
  - `log loss = 1.881`
  - `Brier = 0.810`
  - `mode hit-rate = 45.0%`

Comparado con baseline `alpha = 1.00`:

- `log loss`: `1.974` -> `1.881`
- `Brier`: `0.807` -> `0.810`
- `mode hit-rate`: sin cambio (`45.0%`)

Lectura:

- La mejora fuerte esta en calibracion probabilistica, no en acierto del modo.
- El modelo venia demasiado agresivo; `alpha = 0.55` lo aplana de forma util.
- Se acepta el pequeno coste en `Brier` porque el objetivo prioritario aqui es corregir `log loss`.
