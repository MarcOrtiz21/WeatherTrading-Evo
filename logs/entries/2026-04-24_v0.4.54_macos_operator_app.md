# v0.4.54 - macOS operator app launcher

## Context

Se necesitaba el equivalente macOS de un `.exe`: un lanzador doble-click que cargue la UI operativa, valide el sistema antes de permitir acciones y evite depender de comandos manuales.

## Cambios

- Nuevo builder: `scripts/build_macos_app.py`.
- App generada localmente:
  - `build/macos/WeatherTrading Evo.app`
- El lanzador:
  - valida que existe el proyecto;
  - valida `venv/bin/python`;
  - ejecuta preflight usando `build_operator_dashboard`;
  - abre la UI local;
  - si el preflight esta bloqueado, avisa con alerta macOS y la UI queda en modo lectura.
- La UI ahora incluye `preflight` visible y bloquea `Aprobar paper` si hay blockers criticos.
- `POST /api/decision` tambien respeta el preflight lock.
- `scripts/run_operator_ui.py` detecta si el puerto ya esta ocupado y abre la UI existente en vez de fallar.

## Uso

Construir o reconstruir:

```bash
venv/bin/python scripts/build_macos_app.py
```

Abrir:

```bash
open '/Users/marc/Documents/Polymarket/WeatherTrading/build/macos/WeatherTrading Evo.app'
```

Tambien se puede abrir haciendo doble click en Finder sobre:

```text
/Users/marc/Documents/Polymarket/WeatherTrading/build/macos/WeatherTrading Evo.app
```

## Validaciones realizadas

- Bundle `.app` creado con `Info.plist` valido.
- Ejecutable interno creado en `Contents/MacOS/weathertrading-evo`.
- Lanzamiento mediante `open -n`.
- Backend:
  - `GET /api/health` -> `{"status": "ok"}`
  - `HEAD /` -> `200 OK`
  - `GET /api/dashboard` -> devuelve tickets y preflight
  - `POST /api/export` -> genera `logs/snapshots/2026-04-24_operator_console.json`
- Frontend:
  - UI carga en navegador local.
  - Renderiza preflight/readiness/pipeline/DB/audit.
  - Export muestra toast.
  - Filtro por `Moscow` reduce a `1 visibles / 12 total`.

## Seguridad

El lanzador no activa trading live. Las aprobaciones siguen siendo paper/dry-run locales y se bloquean si el preflight detecta problemas criticos.
