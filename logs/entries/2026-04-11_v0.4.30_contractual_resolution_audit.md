# WeatherTrading v0.4.30 — Contractual Resolution Audit

## Objetivo

Validar si la auditoría ex post basada en `Open-Meteo archive backfill` está midiendo una verdad suficientemente cercana al settlement real de Polymarket, y extraer `evaluate_event` del runner live para poder testearlo de forma aislada.

## Cambios aplicados

- Se ha extraído la lógica de evaluación live a `src/weather_trading/services/evaluation/live_event_evaluator.py`.
- `scripts/run_blind_live_validation.py` ya consume el módulo compartido y deja de mantener una segunda copia local de `evaluate_event` y `build_live_distribution`.
- `src/weather_trading/services/market_discovery/gamma_client.py` conserva metadata contractual útil (`event_id`, `resolution_source_url`, `active`, `closed`, `archived`).
- Se añade `src/weather_trading/services/evaluation/contractual_resolution_validator.py` para comparar:
  - winner Open-Meteo
  - winner contractual Polymarket
  - impacto sobre `paper PnL`
- Se añade `scripts/run_contractual_resolution_audit.py`.
- Se amplían tests unitarios para:
  - `live_event_evaluator`
  - `contractual_resolution_validator`
  - metadata contractual en `gamma_client`

## Interpretación

Este hallazgo cambia la lectura del sistema de forma importante.

- El pipeline ex post con Open-Meteo seguía siendo útil para señal meteorológica y análisis operativo.
- Pero no es una aproximación suficientemente fiable al settlement contractual para hablar de edge económico sin matices.
- La discrepancia no parece ruido menor: hay varios flips de winner por `1-2` bins que cambian el `paper PnL` de forma material.

Conclusión honesta:

- hoy **no** podemos tomar el `paper PnL` acumulado basado en Open-Meteo como proxy económico definitivo;
- a partir de esta versión, la auditoría contractual pasa a ser referencia obligatoria antes de endurecer ejecución o sizing.
