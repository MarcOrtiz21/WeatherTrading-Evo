# v0.4.49 - Operational readiness gate

## Context

Las auditorias y automations recientes confirman que el sistema sigue funcionando a nivel de pipeline, pero no de forma suficientemente fiable para tratarlo como operable:

- el `Daily Pipeline` sigue corriendo
- `blind_live_validation` sigue fallando por red en varios cortes
- la auditoria diaria queda `non_conclusive`
- aun asi, hasta ahora faltaba una capa explicita que dijera si el sistema esta:
  - listo para operar
  - solo apto para paper
  - o directamente bloqueado

## Cambios aplicados

### 1. Readiness gate diario

Se añade:

- `scripts/run_operational_readiness.py`

El script genera:

- `logs/snapshots/<fecha>_operational_readiness.json`

Y clasifica el estado diario en:

- `ready`
- `paper_only`
- `blocked`

Campos principales:

- `status`
- `recommended_mode`
- `pipeline_status`
- `audit_quality`
- `coverage_debt`
- `blockers`
- `warnings`

### 2. Integracion con el pipeline

`scripts/run_daily_pipeline.py` incorpora un nuevo paso:

- `operational_readiness`

De este modo, el pipeline ya no solo produce audit y diagnostico, sino tambien una conclusion operativa diaria reutilizable.

### 3. Reglas iniciales en config

Se añaden en `config/settings.yaml`:

- `block_if_live_snapshot_missing`
- `block_if_audit_non_conclusive`
- `warn_if_audit_partial`

La politica inicial es conservadora:

- sin snapshot live del dia -> `blocked`
- audit `non_conclusive` -> `blocked`
- audit `partial` -> `paper_only`

## Verificacion

- `venv/bin/python -m pytest -q tests/unit/test_operational_readiness.py tests/unit/test_daily_pipeline.py`
- `venv/bin/python -m pytest -q`

## Lectura operativa

Este cambio no activa ejecucion real. Hace algo mas importante primero:

- evita confundir pipeline vivo con sistema operable
- nos obliga a pasar por un gate diario de fiabilidad
- y deja listo el punto donde mas adelante podremos enchufar `paper`, `dry_run` o `live`

## Siguiente paso recomendado

1. Incluir `operational_readiness` en la lectura diaria de automations.
2. No considerar operable ningun dia marcado como `blocked`.
3. Si el readiness empieza a salir `ready` varios dias seguidos, entonces abrir la siguiente capa: sizing, kill switch y paper position tracking diario.
