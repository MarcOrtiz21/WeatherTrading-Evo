# Version 0.4.14

## Contexto

Se prepara una integracion oficial con The Weather Company para poder compararla contra Open-Meteo sobre el mismo formato de mercados Polymarket, especialmente bins e intervalos tipo NYC en Fahrenheit.

La prioridad aqui no es sustituir el pipeline actual a ciegas, sino dejar una capa de prueba limpia para responder dos preguntas:

- si TWC mejora el forecast ex ante en mercados bin/intervalo;
- si su semantica observacional se parece lo suficiente a la contractual de Wunderground como para servirnos en analisis y auditoria.

- `scripts/run_forecast_provider_comparison.py`
- `tests/unit/test_weather_company_client.py`

## Decisiones tecnicas

- Se añade `WeatherCompanyClient` usando la API oficial de `api.weather.com` y key via entorno:
  - `WEATHER_COMPANY_API_KEY`
  - `TWC_API_KEY`
- El cliente devuelve el mismo shape base que ya consume el pipeline con Open-Meteo:
  - `model_max_temp`
  - `model_hourly_temps`
  - `model_hourly_times`
  - `model_cloud_cover_avg`
- El comparador nuevo reutiliza parser, station mapping, distribution builder y pricing ya existentes para comparar proveedores sin rehacer el core de valoracion.
- La comparacion se centra en mercados live de Polymarket con bins/intervalos, no en reverse engineering del frontend de weather.com.
- Se registra `weather_company` en el source registry como fuente oficial de forecast/observacion, pero no se marca como fuente contractual equivalente a Wunderground.

## Resultado practico

- Ya existe una via limpia para probar TWC sin tocar la logica de pricing.
- La integracion no cambia todavia la politica activa ni el proveedor por defecto.
- El sistema ya queda listo para correr comparativas en cuanto haya una API key valida.

## Validacion

- `venv/bin/python -m pytest -q tests/unit/test_weather_company_client.py tests/unit/test_openmeteo_client.py tests/unit/test_parser.py tests/unit/test_pricing_engine.py`
- Resultado: `17 passed`
- `venv/bin/python -m compileall src scripts`

## Riesgos o supuestos

- La comparacion live no pudo ejecutarse desde este sandbox por un fallo de DNS/red en tiempo de ejecucion, no por un error del cliente nuevo.
- Que TWC sea muy fuerte en forecast no implica automaticamente equivalencia con la regla de resolucion contractual de Polymarket/Wunderground.
- Si mas adelante queremos medir mejora ex post real contra ganadores resueltos, necesitaremos validar tambien la disponibilidad de historico de forecast comparable al replay que hoy sacamos de Open-Meteo.

## Siguiente paso recomendado

1. Cargar una key valida de TWC en entorno.
2. Ejecutar `scripts/run_forecast_provider_comparison.py` sobre `NYC,London,Dallas,Helsinki`.
3. Si TWC mejora bins/intervalos de forma consistente, abrir una segunda iteracion para backtest ex post y posible proveedor adaptive.
