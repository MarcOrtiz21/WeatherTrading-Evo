# Version 0.4.12

## Contexto

Esta iteracion refuerza dos capas que ahora mismo son mas importantes que tocar otra vez el forecast:

- fiabilidad operativa
- calidad de evidencia

La idea es que el sistema no solo calcule un edge, sino que tambien sepa si ese edge esta suficientemente respaldado para ser operable y que la auditoria ex post nos diga con claridad cuanta muestra madura pudo resolverse de verdad.

## Decisiones tecnicas

- Se crea `OperationalEvidenceGate` como capa reusable para puntuar la solidez operativa de cada evento.
- La evaluacion de evidencia combina:
  - confianza del parser
  - profundidad de calibracion
  - miembros de ensemble disponibles
  - dispersion prevista
  - contexto intradia
- Se anade configuracion nueva en `operational_evidence`:
  - `min_event_score`
  - `min_parser_confidence`
  - `min_calibration_days_for_calibrated`
  - `max_forecast_std_dev_c`
  - `max_late_intraday_remaining_hours_without_observed_max`
- Para eventos del mismo dia, el sistema ya no acepta sin mas un proxy intradia tarde si no hay maximo observado real; eso puede bloquear el evento con `late_intraday_without_observed_max`.
- En la validacion live, los `blockers` de evidencia se agregan a cada mercado. Si la evidencia del evento es floja, el mercado deja de ser operable aunque el quote sea bueno.
- Los snapshots live nuevos guardan metadata de evidencia por evento:
  - `event_operable`
  - `event_evidence_score`
  - `event_evidence_tier`
  - `event_blockers`
  - `event_evidence_notes`
  - `parser_confidence_score`
  - `station_region`
- La auditoria ciega ahora incluye una capa de cobertura:
  - `mature_events`
  - `mature_resolution_coverage`
  - `mature_skip_rate`
  - `skip_reason_counts`
  - `pending_reason_counts`
- La evaluacion ex post tambien se desglosa por:
  - `event_evidence_tier`
  - `actual_temperature_source`

## Resultado practico

- El sistema ya no mezcla "hay edge" con "hay suficiente respaldo para actuar".
- Los eventos con evidencia debil pueden seguir apareciendo en snapshots, pero quedan claramente marcados como no operables.
- La auditoria deja de ser solo una tabla de aciertos y errores; ahora tambien mide cobertura real de resolucion y enseña si el bloqueo viene por falta de observacion local, archivo remoto no disponible o problemas de catalogo.

## Riesgos o supuestos

- El score de evidencia es una heuristica operativa, no una verdad estadistica final.
- Bloquear eventos tardios sin observacion local puede reducir oportunidades aparentes, pero sube disciplina y evita sobreconfiar en proxies.
- Los snapshots antiguos no traen esta metadata nueva; la auditoria hace fallback razonable cuando falta.

## Siguiente paso recomendado

1. Dejar correr varios dias esta version para ver cuantos eventos quedan operables de verdad.
2. Revisar las nuevas auditorias para comprobar si mejora la relacion entre `paper PnL` y evidencia alta.
3. Priorizar la carga de observaciones locales reales en `weather_trading.db`, porque esa es ahora la pieza que mas puede destrabar la calidad de evidencia ex post.
