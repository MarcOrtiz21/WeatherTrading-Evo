# v0.4.59 - H1-only operator policy

## Context

The post-fix 2026-05-01 audit confirmed that the operational edge remains concentrated in H1 while H0 continues to damage calibration and paper returns:

- H0: ROI -19.2%, model log loss 5.582 vs market 2.093.
- H1: ROI +47.7%, model log loss 1.689 vs market 1.246.

The operator was already blocking H0 via intraday guardrails, but the rule needed to be explicit in the policy layer and visible in the console/UI.

## Changes

- Added `operator_policy` to `config/settings.yaml`.
- Formalized `min_trade_horizon_days: 1`, making H1+ the minimum reviewable trade horizon.
- Added explicit `below_min_trade_horizon` ticket blocker for H0 tickets.
- Added an execution policy summary to the operator console payload.
- Added text output showing `Policy: H1+ only | H0=quarantined | copytrading=veto_only`.
- Added a Policy KPI to the Operator UI.
- Added tests covering the H1-only policy summary and H0 blocker.

## Result

The system now communicates the operating stance directly:

- H0 is quarantined.
- H1+ is the only reviewable horizon.
- Copytrading remains veto/context only.
- Live execution remains disabled.

This makes the daily review safer: if the live market list is all H0, the expected outcome is `NO_TRADE`, not a confusing lack of approvals.
