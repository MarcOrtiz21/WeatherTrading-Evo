# v0.4.57 - Launchd health and horizon-0 quarantine

## Context

The 2026-04-29 operator review showed a healthy manually recovered daily pipeline, but the macOS LaunchAgents were still failing with exit code 127 before entering Python. The same audit also showed that horizon-0 markets are materially underperforming the rest of the system:

- Overall sample: 260 evaluated events, model mode hit-rate 48.5% vs market 44.6%.
- Overall log loss: model 2.563 vs market 1.462.
- H0: ROI -22.1%, model log loss 5.477 vs market 2.180.
- H1: ROI +47.7%, model log loss 1.689 vs market 1.246.

## Changes

- Added real LaunchAgent inspection to the operator console health report, including status, run count, last exit code, and stderr tail.
- Added a warning when the project root is inside macOS protected user folders such as `Documents`, because LaunchAgents may be blocked by privacy/TCC restrictions.
- Exposed horizon-level audit summaries in the operator console, with explicit H0 log-loss and ROI warnings.
- Changed the default `operator_risk.horizon0_intraday_policy` from `cap` to `block`, quarantining same-day intraday tickets until a separate H0 policy earns its way back in.
- Updated the Operator UI scheduler card to surface failed LaunchAgents directly.
- Added tests for failed LaunchAgent detection and H0 audit deterioration.

## Result

The 2026-04-29 operator console now reports:

- `launchd_scheduler_not_healthy`
- `project_root_in_macos_protected_directory`
- `horizon0_log_loss_underperforms_market`
- `horizon0_negative_paper_roi`
- all current H0 tickets blocked with `same_day_intraday_blocked`
