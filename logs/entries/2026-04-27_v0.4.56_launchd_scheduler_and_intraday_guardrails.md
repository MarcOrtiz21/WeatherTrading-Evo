# WeatherTrading v0.4.56 - Launchd scheduler and intraday guardrails

## Context

The `2026-04-27` Codex automations did not generate daily artifacts by mid-afternoon, but a manual recovery run completed successfully. This confirms the pipeline itself works and the weak point is scheduler reliability.

The newest mature cohort also showed severe same-day deterioration:

- Horizon `0` model log loss: `5.536` vs market `2.072`
- Horizon `0` paper ROI: `-8.3%`
- Cohort `2026-04-26` paper ROI: `-74.2%`

## Changes

- Added `scripts/run_daily_pipeline_launchd.sh` as a macOS `launchd` runner with lock protection and dated logs.
- Added `scripts/install_launchd_scheduler.py` to install two LaunchAgents:
  - `com.weathertrading.evo.daily-pipeline` at `08:20` Europe/Madrid.
  - `com.weathertrading.evo.pipeline-watchdog` at `09:10` Europe/Madrid.
- Installed and loaded both LaunchAgents under `~/Library/LaunchAgents`.
- Added same-day intraday guardrails in the operator console:
  - Cap horizon `0` exposure to max `$1`.
  - Apply a `0.20x` same-day intraday stake multiplier.
  - Block same-day tickets when too many forecast hours remain.
  - Block late same-day tickets without local observed max evidence.
- Added unit coverage for LaunchAgent plist generation and same-day intraday ticket blocking/capping.

## Result

The `2026-04-27` operator console now blocks all current same-day candidates because they are still too early intraday. This is intentional after the observed horizon `0` degradation.

The `launchd` scheduler is installed and visible via:

```bash
launchctl print gui/$(id -u)/com.weathertrading.evo.daily-pipeline
launchctl print gui/$(id -u)/com.weathertrading.evo.pipeline-watchdog
```
